## Initial Analysis
The given challenge is a 64-bit ELF executable. Running it directly or in a debugger causes it to crash with `SIGSEGV` or `SIGILL`. 
A quick static analysis reveals a string `apoorvctf{wh4t_1f_k1ng_cr1ms0n_requ13m3d??}`. Submitting this flag fails, meaning this is a decoy flag.

## Anti-Debugging & Obfuscation
Using `strace -f ./challenge`, we observe the program spawns multiple threads. Thread 1 intentionally crashes with `SIGILL` using a `ud2` instruction after performing AES encryption on the fake flag. The AES operations serve as a red herring.

Thread 2 handles the actual flag generation but uses a **signal-oriented instruction stream**: it intentionally causes `SIGSEGV`s and does its logic inside the segmentation fault handler. 

To determine when we are being debugged, the program uses the `rdtsc` instruction in the `SIGSEGV` handler. If the time difference between exceptions is too large (which happens when tracing in GDB), it sets a penalty variable `[0x54991]` to `1`.

## Thread 3 (The Eraser)
Thread 3 implements the "King Crimson" (erasing time) effect described in the challenge description. It waits for the flag buffer to be filled. As soon as the true flag is fully decrypted, Thread 3 wakes up, zeroes out the flag buffer, and `munmap`s the memory region before the main thread can print it.

This is why, even if you manage to bypass the debugger checks, the buffer prints empty.

## Extracting the True Flag
We discovered that the true flag is generated byte-by-byte in a Mixed Boolean Arithmetic (MBA) sequence located in Thread 2's `SIGSEGV` handler (`b583` and `b5e3`). 

The calculation depends on:
1. `b583(i)`: A custom MBA evaluation based on the index `i` (0 to 39).
2. `b5e3(i)`: An S-Box substitution operation and bit-shift.
3. `[rodata]`: Hardcoded expected bytes from `.rodata` offset `0x45c1c` and `0x45d30`.
4. The Anti-Debug Penalty.

When the anti-debug time check fails, the program adds `0x37` to the generated bytes:
```assembly
    84 c9                	test   cl,cl
    0f b6 c8             	movzx  ecx,al
    0f b6 c2             	movzx  eax,dl   <-- (eax = al + 0x37)
    0f 44 c1             	cmove  eax,ecx  <-- If cl (penalty) is 0, use proper bytes
```

To bypass both the S-Box obfuscation and the `rdtsc` checks, the MBA logic was ported from assembly into Python. By evaluating the pure mathematical operations and avoiding the `0x37` penalty, we can derive the correct bytes!

### Python Solver
```python
def b519(edi, esi): return (edi + esi) & 0xff
def b53c(edi, esi): return (edi ^ esi) & 0xff

def b583(edi):
    val1 = (edi + edi) & 0xff
    val2 = (val1 + val1) & 0xff
    val3 = (val2 + val1) & 0xff
    val4 = (val3 + edi) & 0xff
    val5 = (val4 + 0x3f) & 0xff
    bl = ((edi << 3) | (edi >> 5)) & 0xff
    return b53c(val5, bl)

SBOX = [0x6a09e667, 0xbb67ae85, 0x3c6ef372, 0xa54ff53a, 0x510e527f, 0x9b05688c, 0x1f83d9ab, 0x5be0cd19]
def b5e3(edi):
    idx1 = edi & 7
    val = SBOX[idx1]
    shift = (edi >> 3) & 3
    cl = shift << 3
    return (val >> cl) & 0xff

rodata1 = bytes.fromhex("39407691b717c97879013adf3a2adea11c2b04e0")
rodata2 = bytes.fromhex("bb19b025e37eaa786c4116e7aeea00c9c623940d")

flag = []
for i in range(40):
    val_583 = b583(i)
    val_e3 = b5e3(i)
    
    r12 = i // 2
    if i % 2 == 0:
        mem = rodata1[r12]
    else:
        mem = rodata2[r12]
        
    bpl = val_583 ^ val_e3 ^ mem
    flag.append(chr(bpl))

print("TRUE FLAG:", "".join(flag))
```

Running the solver outputs the true flag perfectly without the penalty offset.
